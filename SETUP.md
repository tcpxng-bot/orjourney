# คู่มือติดตั้ง OR Journey (ฉบับใช้งานจริง)

ทำตามลำดับ 1 → 5 ครั้งเดียวจบ ใช้เวลาประมาณ 30–45 นาที

> **สำคัญ:** อย่าเพิ่งใช้กับผู้ป่วยจริงจนกว่าจะทดสอบครบตามข้อ 5 แล้ว

---

## 1. เตรียมฐานข้อมูล (Supabase)

เปิดโปรเจกต์ Supabase ของคุณ → เมนู **SQL Editor** → กด **New query**
แล้วรันไฟล์ตามลำดับนี้ **ทีละไฟล์** (คัดลอกเนื้อไฟล์ไปวางแล้วกด Run)

| ลำดับ | ไฟล์ | ได้อะไร |
|------|------|---------|
| 1 | `sql/01_schema.sql` | ตาราง 11 ตาราง, สถานะ, ทริกเกอร์ |
| 2 | `sql/02_rls.sql` | สิทธิ์การเข้าถึง (RLS) + ฟังก์ชันค้นหาสำหรับญาติ |
| 3 | `sql/03_seed.sql` | ข้อมูลตั้งต้น: วอร์ด, ห้อง OR, อวตาร์ 12 ตัว |

ถ้ารันแล้วขึ้น error ให้หยุดก่อนแล้วแก้ให้ผ่านทีละไฟล์ อย่าข้ามไปไฟล์ถัดไป

**เปิด Realtime:** เมนู **Database → Replication** → เปิด (toggle) ตาราง `journeys`
ถ้าไม่เปิด หน้าจอจะไม่อัปเดตอัตโนมัติ ต้องรีเฟรชเอง

---

## 2. สร้างบัญชีผู้ใช้และกำหนดบทบาท

**2.1 สร้างผู้ใช้** — เมนู **Authentication → Users → Add user**
กรอกอีเมลกับรหัสผ่าน แล้วติ๊ก **Auto Confirm User** (ไม่งั้นต้องยืนยันอีเมลก่อนถึงจะเข้าได้)

สร้างอย่างน้อย 5 บัญชีสำหรับทดสอบ เช่น
`ward@test.local` · `porter@test.local` · `or@test.local` · `rr@test.local` · `admin@test.local`

**2.2 กำหนดบทบาท** — ระบบจะสร้างแถวใน `profiles` ให้อัตโนมัติ (บทบาทเริ่มต้นคือ `WARD`)
ไปที่ **SQL Editor** แล้วรันคำสั่งนี้ โดยแก้อีเมลให้ตรงกับที่สร้างไว้

```sql
-- ดูรายชื่อผู้ใช้และ id ก่อน
select p.id, u.email, p.role, p.ward_id, p.full_name
from profiles p join auth.users u on u.id = p.id;

-- กำหนดบทบาท + ชื่อที่จะแสดงตอนยืนยันป้ายข้อมือ
update profiles set role='PORTER', full_name='คุณสมชาย'
  where id = (select id from auth.users where email='porter@test.local');

update profiles set role='OR', full_name='พว. สมหญิง'
  where id = (select id from auth.users where email='or@test.local');

update profiles set role='RR', full_name='พว. ปรียา'
  where id = (select id from auth.users where email='rr@test.local');

update profiles set role='ADMIN', full_name='ผู้ดูแลระบบ'
  where id = (select id from auth.users where email='admin@test.local');

-- บัญชีหอผู้ป่วย ต้องผูกกับวอร์ดด้วย ไม่งั้นจะสร้าง Journey ไม่ได้
update profiles set role='WARD', full_name='พว. หอสูติ',
       ward_id = (select id from wards where name='สูตินรีเวชกรรม' limit 1)
  where id = (select id from auth.users where email='ward@test.local');
```

บทบาทที่ใช้ได้: `WARD` · `PORTER` · `OR` · `RR` · `PR` · `ADMIN`

> **ข้อควรระวัง:** `full_name` คือชื่อที่จะถูกบันทึกลงประวัติการยืนยันป้ายข้อมือ
> ควรใส่ชื่อจริงของเจ้าหน้าที่ เพราะใช้ตรวจสอบย้อนหลังได้

## 3. ใส่ค่าเชื่อมต่อในเว็บ

เปิดไฟล์ `js/config.js` แล้วกรอก 2 ค่า หาได้จาก Supabase → **Settings → API**

```js
window.OJ_CONFIG = {
  supabaseUrl:     'https://xxxxxxxx.supabase.co',
  supabaseAnonKey: 'eyJhbGciOi...',
  forceDemo: false,
};
```

> **ห้ามใส่ `service_role` key เด็ดขาด** ไฟล์นี้ทุกคนที่เปิดเว็บโหลดไปดูได้
> ใส่ได้แค่ **anon key** เท่านั้น ความปลอดภัยจริงมาจาก RLS ในฐานข้อมูล

ถ้าเว้นว่างทั้งสองค่า เว็บจะรันเป็น**โหมดเดโม** (ข้อมูลจำลอง) ซึ่งใช้ซ้อมมือได้

## 4. เอาขึ้นเว็บ (Vercel)

**4.1** สร้าง repository ใน GitHub แล้ว push โฟลเดอร์นี้ขึ้นไป

**4.2** เข้า [vercel.com](https://vercel.com) → **Add New → Project** → เลือก repository

**4.3** ตั้งค่าตอน import
- Framework Preset: **Other**
- Build Command: *(เว้นว่าง)*
- Output Directory: *(เว้นว่าง)*

เว็บนี้เป็น static ล้วน ไม่ต้อง build

**4.4** กด **Deploy** รอสักครู่จะได้ลิงก์ เช่น `or-journey.vercel.app`

**4.5** กลับไปที่ Supabase → **Authentication → URL Configuration**
ใส่ลิงก์ Vercel ลงใน **Site URL** และ **Redirect URLs**

## 5. ทดสอบก่อนใช้จริง (อย่าข้าม)

ทดสอบครบทุกข้อ ติ๊กไปทีละข้อ

- [ ] เปิดเว็บแล้วเห็นหน้าล็อกอิน (ไม่ใช่หน้าเลือกบทบาทแบบเดโม)
- [ ] ล็อกอินผิดรหัส → ขึ้นข้อความ "อีเมลหรือรหัสผ่านไม่ถูกต้อง"
- [ ] ล็อกอินบัญชีวอร์ด → เห็นชื่อหอผู้ป่วยของตัวเอง กดสร้าง Journey ได้ ได้บัตรอวตาร์
- [ ] ล็อกอินบัญชีหน่วยเปล → เห็นเคสในคิว "รอไปรับ" กดรับ + ติ๊กป้ายข้อมือ + เลือกห้องได้
- [ ] ล็อกอินบัญชี OR → ยืนยัน 2 พยาบาล (ต้องคนละคน) → เริ่มผ่าตัด → ผ่าตัดเสร็จ
- [ ] ล็อกอินบัญชี RR → รับเข้า + ติ๊กป้ายข้อมือ → ส่งกลับหอ
- [ ] เปิดสองเครื่องพร้อมกัน กดที่เครื่องหนึ่ง **อีกเครื่องต้องเปลี่ยนตามภายในไม่กี่วินาที**
- [ ] บัญชีวอร์ด A ต้อง **ไม่เห็น** เคสของวอร์ด B
- [ ] ทดสอบเปิดเคสฉุกเฉินจากบัญชี OR (เลือกต้นทางเป็นห้องคลอด) → ต้องเด้งขึ้นบนสุดของคิวหน่วยเปล
- [ ] ลองค้นหาสถานะแบบญาติ (บทบาท PR) ด้วยอวตาร์ + รหัส
- [ ] เคสที่จบแล้ว ค้นด้วยรหัสเดิมต้อง **ไม่เจอ**

**ถ้าข้อไหนไม่ผ่าน** เปิด Console ของเบราว์เซอร์ (F12 → Console) จะเห็นข้อความขึ้นต้นด้วย
`[OR Journey]` บอกว่าติดตรงไหน สาเหตุที่พบบ่อยคือ RLS ไม่อนุญาต หรือยังไม่ได้กำหนดบทบาท

## ปัญหาที่พบบ่อย

| อาการ | สาเหตุ | วิธีแก้ |
|------|--------|--------|
| ขึ้น "ไม่พบโปรไฟล์ผู้ใช้" | ทริกเกอร์สร้าง profile ไม่ทำงาน | รัน `01_schema.sql` ใหม่ แล้วลบ-สร้างผู้ใช้ใหม่ |
| ขึ้น "บัญชีนี้ยังไม่ได้กำหนดบทบาท" | ยังไม่ได้ทำข้อ 2.2 | รัน `update profiles set role=...` |
| วอร์ดกดสร้างไม่ได้ | บัญชียังไม่ผูก `ward_id` | รัน `update profiles set ward_id=...` |
| หน้าจอไม่อัปเดตเอง | ยังไม่เปิด Realtime | Database → Replication → เปิดตาราง `journeys` |
| ขึ้น "ไม่มีสิทธิ์ทำรายการนี้" | RLS ทำงานถูกต้องแล้ว | ตรวจว่าบทบาทตรงกับขั้นตอนที่กด |
| ชื่อหอผู้ป่วยขึ้นเป็น "—" | ยังไม่ได้รัน `03_seed.sql` | รัน seed แล้วรีเฟรชหน้าเว็บ |
| หน้าเว็บว่างเปล่า | ไฟล์ js โหลดไม่ครบ | ตรวจว่า push โฟลเดอร์ `js/` และ `css/` ขึ้นครบ |

## หมายเหตุด้านความปลอดภัยและ PDPA

- ระบบนี้ **ไม่มีคอลัมน์สำหรับเก็บชื่อผู้ป่วย HN เลขบัตร การวินิจฉัย หรือชื่อหัตถการ** — เก็บไม่ได้แม้จะอยากเก็บ
- การยืนยันป้ายข้อมือบันทึกเพียงว่า *ยืนยันแล้ว* และ *ใครยืนยัน* ไม่เก็บชื่อผู้ป่วย
- ญาติเข้าถึงผ่านฟังก์ชันเดียวที่ปลอดภัย และทุกครั้งถูกบันทึก audit
- รหัสติดตามเป็นตัวช่วยความสะดวก ไม่ใช่ความลับระดับสูง — ควรแนะนำญาติไม่ให้เผยแพร่
- รายละเอียดเพิ่มเติมดูที่ `docs/SECURITY.md`

**ก่อนใช้งานจริงกับผู้ป่วย** ควรผ่านการพิจารณาของคณะกรรมการสารสนเทศ/ความเป็นส่วนตัวของโรงพยาบาลก่อน
